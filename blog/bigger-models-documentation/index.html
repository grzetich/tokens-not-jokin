The Surprising Thing About Bigger AI Models and Documentation

Cloud AI models pass every documentation format test I threw at them. 100% across the board. That sounds like good news. It isn't.

The result that almost fooled me

When I ran my first round of tests with Claude Sonnet and GPT-4o, the results looked boring. Every format, every API, every endpoint: 100% pass rate. YAML, OpenAPI, Markdown, DON. Didn't matter. The big models ate it all and produced working code every time.

If I had stopped there, the conclusion would have been simple: format doesn't doesn't affect code quality or output. Use whatever format for your API docs you want. And that would have been completely wrong.

What the big models are hiding

A 100% pass rate across all formats doesn't mean all formats are equal. It means the model is powerful enough to compensate for format inefficiency. It's brute-forcing through verbose documentation the same way a fast CPU can brute-force through unoptimized code. The code works, so nobody notices the waste.

But the waste is real. A verbose format consumes more of the model's context window just to describe the same API. That's context that could be used for the developer's project files, for reasoning about edge cases, or for generating more complete code. When the documentation is the only thing in context, the big model handles it fine. In a real workflow, with other information competing for that same context window, the tax adds up.

YAML describes the same API information using up to 80% fewer tokens than OpenAPI 3.0. That's not a rounding error. That's the difference between your documentation leaving room for the model to think, or filling the window before it starts.

Where the truth shows up

The real story emerged when I tested smaller models. Not toy models, but real open-source models from Alibaba and Microsoft: a 7B parameter model and a 3.8B parameter model. The kind of models that run locally on a laptop.

On the simple control API (6 endpoints), even the small models handled most formats well. OpenAPI dropped to 83%, but everything else stayed at 100%. Format mattered, but not dramatically.

Then I tested the complex control API. Ten endpoints, authenticated webhook verification, cursor-based pagination, partial updates. The kind of API patterns that exist in the real world.

You can read about both APIs in more detail in this blog post, and of course, in the book.

The 3.8B model with YAML: 100% pass rate. With Markdown: 100%. With DON: 100%.

With OpenAPI: 0%. Not low. Zero. The model didn't produce bad code. It stopped producing code entirely. The failures weren't wrong functions with bad parameters. They were fragments of commas and quotes. The 44,000-character OpenAPI spec went in, and the model couldn't generate coherent output of any kind.

Same information, different outcome

This is the part that matters: all four formats describe the exact same API. Same endpoints, same parameters, same constraints. The only thing that changed was how that information was structured. And that structural choice determined whether a small model could use the API at all.

The big models masked this entirely. If you only test with a frontier model, you'll never see it. Your docs look fine. Your tests pass. And then a developer using a local model, or a model with a smaller context window, or a model that's also holding their project files and conversation in context, hits a wall you didn't know existed.

The Invisible Wall

Even when the big models pass every test, format still affects what happens on the other side. A model reading a 44,000-character OpenAPI spec has less room for everything else than a model reading an 8,500-character YAML spec that describes the same API. Less room for the developer's project files, less room for reasoning, less room for generating complete code. Your docs work. They could just work better, and you'd never know the difference from your side.

This is why I call it the invisible wall. The big model compensates. The code works. Nobody complains. But the model is spending capacity on parsing verbose documentation instead of writing better code. Your customers are paying for it in tokens, and developers are paying for it in quality they never see because they have nothing to compare it against.

What this means

If you maintain API documentation, the big model pass rates are misleading. They tell you everything works, which is true…and useless. The interesting question isn't whether Claude can handle your docs. It's whether your docs are making Claude work harder than it needs to, and whether a developer with a smaller model or a fuller context window will hit a wall you didn't know existed.

The format of your documentation is the variable you control. Model capability is not. And in my data, format explains over 10x more variance in generated code than model choice.

Most of the industry conversation is about which AI model is best. The data says the format of your documentation matters more for what that model produces.
