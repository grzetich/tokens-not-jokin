Coming in May: HTML Format Testing

The book tested YAML, OpenAPI, DON, and Markdown. But most API documentation ships as HTML. That gap is getting closed.

The format we didn't test

The original research in Tokens Not Jokin' compared four documentation formats: YAML, OpenAPI 3.0, DON, and Markdown. Those four covered the structured spec formats that teams choose between when deciding how to represent their API.

But they don't cover how most developers actually encounter API documentation. Most API docs ship as rendered HTML pages, complete with navigation, sidebars, code examples, syntax highlighting, and all the markup that makes a developer portal readable. When an AI coding tool fetches documentation from a URL, it often gets that full HTML page, not a clean spec file.

That means the most common delivery format for API documentation wasn't in the original dataset. In May, it will be.

What the tests will cover

I'm adding verbose HTML as a fifth format to the existing test suite. Same two control APIs (BookClub and EventForge), same models, same tasks, same methodology. The HTML versions will include the kind of markup a typical developer portal produces: navigation elements, formatting tags, code blocks with syntax highlighting wrappers, meta content, and the structural overhead that comes with rendering docs for a browser.

The question is straightforward: how does a rendered HTML API doc page perform compared to YAML, OpenAPI, Markdown, and DON when an AI model uses it to generate code? Does the extra markup hurt token efficiency? Does it affect pass rates? Does it change code patterns?

Based on the existing data, verbose HTML will almost certainly use more tokens than any of the four original formats. The interesting question is whether that token overhead affects code generation quality the way OpenAPI's verbosity did on smaller models.

Why this matters now

Tools like Mintlify, GitBook, and ReadMe are building infrastructure to serve documentation to AI tools. Approaches like llms.txt exist specifically to give agents a cleaner version of docs than the full HTML page. My assumption behind all of that work is that raw HTML is a bad format for AI consumption. The research will put numbers on that assumption.
